# SonarQube MR 流水线说明

> 面向：无 SonarQube / CI 基础的开发人员

---

## 📌 这套流水线是做什么的？

当你发起一个 **合并请求（MR）** 时，这套流水线会自动：

1. **扫描代码质量** — 检查有没有 Bug、漏洞、代码异味
2. **生成报告** — 输出 PDF 格式的分析报告
3. **评论到 MR** — 在你的合并请求下自动留言，告诉你扫出了多少问题
4. **质量门检查** — 如果问题太多，直接阻断合并
5. **自动合并（仅 UAT）** — UAT 分支质量门通过后，自动合并代码

---

## 🏗️ 流水线架构

```
┌─────────────┐     ┌─────────────┐
│  analysis   │ ──► │    merge    │
│   阶段      │     │   阶段      │
└─────────────┘     └─────────────┘
      │                   │
      ▼                   ▼
  代码扫描           MR 合并
  生成 PDF          (仅 UAT)
  发评论/通知
```

**两个环境区别：**

| 环境 | 扫描 | 合并方式 |
|------|------|----------|
| **UAT** | ✅ 自动 | ✅ 自动（质量门通过后） |
| **Master** | ✅ 自动 | ⏸️ 手动（需人点按钮） |

---

## 🔍 analysis 阶段做了什么？

### 1. 分支规则校验

根据你合并的目标分支，检查源分支命名是否合规：

| 目标分支 | 允许的源分支 |
|----------|-------------|
| `uat` | `feature/*`、`hotfix/*` |
| `master` | `release/*` |

**示例：**
- ✅ `feature/login` → `uat` ✅ 允许
- ❌ `bugfix/login` → `uat` ❌ 不允许，会直接失败

### 2. SonarQube 代码扫描

调用 SonarQube 对 **本次 MR 新增的代码** 进行分析，检查三类问题：

| 类型 | 含义 | 严重程度 |
|------|------|----------|
| 🐞 Bugs | 代码错误 | 阻塞 > 严重 > 主要 > 次要 > 提示 |
| 🔒 Vulnerabilities | 安全漏洞 | 同上 |
| 💨 Code Smells | 代码异味（可优化） | 同上 |

同时还会检查：
- **可靠性评级**（A-E，A最好）
- **安全性评级**（A-E）
- **可维护性评级**（A-E）
- **覆盖率**（单元测试覆盖率）

### 3. 自动评论到 MR

扫描完成后，Bot 会在你的 MR 下自动留言，格式如下：

```
SonarQube Quality Gate: ✅ 通过 / ❌ 未通过

新增问题统计（本次MR）:
- 新增 Bugs: 3
- 新增 Vulnerabilities: 1
- 新增 Code Smells: 15

质量门评级:
- 可靠性: B
- 安全性: A
- 可维护性: C
- 覆盖率: 65%

[查看 SonarQube 分析](链接)
```

### 4. 生成 PDF 报告

自动生成一份**中文 PDF 报告**，包含：
- 项目信息、MR 信息、分支信息
- 质量门状态
- 问题统计卡片
- Bug / 漏洞 / 异味的详细列表（每个列出文件名、行号、严重性、描述）

PDF 上传到 **MinIO 文件存储**，并在 MR 下添加下载链接。

### 5. 钉钉通知（失败时）

如果质量门**未通过**，会自动发钉钉消息通知相关人。

---

## 🔀 merge 阶段做了什么？

### UAT 环境（自动合并）

质量门通过后，**自动**执行合并：

```
✅ 状态检查 → MR 状态为 opened → 可以合并 → PUT /merge
```

### Master 环境（手动合并）

流水线不会自动合并，需要负责人**手动点击** "Play" 按钮触发合并。

---

## ⛔ 什么情况下会阻断合并？

1. **分支命名不合规** — 如 `bugfix/xxx` 合并到 `uat`
2. **质量门未通过** — SonarQube 判定问题超标
3. **合并时 MR 状态异常** — 如已被关闭、或有冲突

---

## 📋 分支命名规范

```
feature/<功能名>    → 合并到 uat ✅
hotfix/<修复名>     → 合并到 uat ✅
release/<版本名>    → 合并到 master ✅
其他                → 不允许 ❌
```

## ❓ 常见问题

**Q: 为什么我的 MR 合并失败了？**
A: 检查 MR 下的 Bot 评论，会说明具体原因（命名不合规 / 质量门未通过）

**Q: 质量门是什么？**
A: SonarQube 定义的一套代码质量标准，达到标准才允许合并

**Q: PDF 报告在哪里？**
A: MR 评论里有点击链接，也在钉钉通知里

**Q: master 分支合并需要手动吗？**
A: 是的，需要负责人点击 GitLab 流水线上的 "Play" 按钮
