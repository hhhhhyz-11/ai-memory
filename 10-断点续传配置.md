# KingbaseES V9R2C14 断点续传配置与验证

> 适用版本：KingbaseES V9R2C14
> 参考：https://docs.kingbase.com.cn/cn/KES-V9R2C14/sync/

---

## 一、断点续传原理

### 1.1 什么是断点续传

KFS 在同步过程中，如果发生网络中断、目标端故障或服务重启，可以从**上次中断的位置**继续同步，而不需要重新全量同步。

### 1.2 实现机制

断点续传依赖以下机制：
1. **复制槽（Replication Slot）** — 保存 WAL 位置信息
2. **检查点（Checkpoint）** — 定期记录同步状态
3. **LSN（Log Sequence Number）** — WAL 日志序列号

## 二、配置断点续传

### 2.1 前提条件

```sql
-- 1. 源端必须启用复制槽
-- kingbase.conf 配置：
wal_level = replica
max_replication_slots >= 5
wal_keep_size >= 1GB
```

### 2.2 KFS 配置启用断点续传

```bash
# 在 KFS 配置文件中启用
cat >> conf/目标端.properties << 'EOF'

# === 断点续传配置 ===
replication.checkpoint.enabled=true
replication.checkpoint.interval=30

# === 复制槽配置 ===
replication.slots=true
replication.slot.name=flysync_slot
replication.slot.keepalive=true

# === 自动重连 ===
reconnection.maxRetries=10
reconnection.retryInterval=30
reconnection.enabled=true
EOF
```

### 2.3 检查复制槽是否正常

```sql
-- 源端查询复制槽状态
SELECT
    slot_name,
    slot_type,
    active,
    restart_lsn,
    confirmed_flush_lsn,
    pg_size_pretty(pg_wal_lsn_diff(pg_current_wal_lsn(), restart_lsn)) AS lag
FROM sys_replication_slots
WHERE slot_name LIKE 'flysync%';
```

## 三、模拟断点续传测试

### 3.1 测试场景

```
正常同步 → 网络中断 → KFS 重连 → 断点续传 → 继续同步
```

### 3.2 步骤 1：正常同步

```bash
# 启动 KFS 同步
bin/flysync start -conf conf/目标端.properties -dir data/target

# 记录同步位置
bin/flysync status -dir data/target -format json | grep -E "lsn|position"
```

### 3.3 步骤 2：模拟网络中断

```bash
# 方法 1：停止目标端 KFS
bin/flysync stop -dir data/target

# 方法 2：使用 iptables 模拟网络中断
iptables -A INPUT -s 192.168.1.101 -j DROP
# ... 等待一段时间 ...
iptables -D INPUT -s 192.168.1.101 -j DROP
```

### 3.4 步骤 3：验证续传

```bash
# 重启 KFS（会自动从断点续传）
bin/flysync start -conf conf/目标端.properties -dir data/target

# 检查状态
bin/flysync status -dir data/target

# 检查日志确认续传
grep -i "resume\|recover\|restart" /opt/kingbase-flysync/logs/flysync-*.log
```

## 四、验证断点续传

### 4.1 检查点日志

```bash
# 查看 KFS 检查点日志
grep -i checkpoint /opt/kingbase-flysync/logs/flysync-*.log

# 典型输出：
# 2026-04-07 10:30:00 [INFO] Checkpoint completed: lsn=0/12345678, offset=12345
# 2026-04-07 10:30:30 [INFO] Checkpoint completed: lsn=0/12345700, offset=12400
```

### 4.2 查看当前同步位置

```sql
-- 源端：查看 WAL 发送位置
SELECT
    application_name,
    state,
    sent_lsn,
    write_lsn,
    flush_lsn,
    replay_lsn,
    pg_wal_lsn_diff(pg_current_wal_lsn(), replay_lsn) AS lag_bytes
FROM sys_stat_replication;

-- 目标端：查看最后应用的 LSN
SELECT
    last_wal_receive_lsn,
    last_wal_replay_lsn,
    pg_wal_lsn_diff(pg_current_wal_lsn(), last_wal_replay_lsn) AS lag_bytes,
    last_xact_replay_timestamp
FROM sys_stat_wal_receiver;
```

### 4.3 对比源端和目标端数据

```sql
-- 源端记录数
SELECT count(*) FROM sync_table WHERE update_time > '2026-04-07 00:00:00';

-- 目标端记录数
SELECT count(*) FROM sync_table WHERE update_time > '2026-04-07 00:00:00';

-- 差值应该为 0（续传成功后）
```

## 五、常见问题

### 5.1 续传失败：复制槽不存在

```
症状：KFS 重启后报错 replication slot does not exist
原因：复制槽被删除或 wal_keep_size 太小导致 WAL 被清理
```

**解决方案：**

```sql
-- 检查复制槽
SELECT * FROM sys_replication_slots WHERE slot_name = 'flysync_slot';

-- 如果不存在，重新创建
SELECT * FROM sys_create_physical_replication_slot('flysync_slot', true);

-- 或者在 KFS 配置中使用 auto_slot=true 自动创建
```

### 5.2 续传失败：WAL 被清理

```
症状：restart_lsn 之后的 WAL 已经不存在
原因：wal_keep_size 太小，或复制槽未启用
```

**解决方案：**

```bash
# 增加 wal_keep_size
sed -i "s/wal_keep_size = .*/wal_keep_size = 4GB/" /data/kingbase/data/kingbase.conf

# 或使用复制槽（推荐）
# kingbase.conf 中已有：
# max_replication_slots = 10

# 重载配置
./sys_monitor reload -D /data/kingbase/data
```

### 5.3 续传延迟过大

```bash
# 检查网络带宽
iperf3 -c <peer_ip>

# 增加 KFS 并行度
sed -i "s/replication.parallelism=.*/replication.parallelism=8/" conf/目标端.properties

# 增加批处理大小
sed -i "s/replication.batchSize=.*/replication.batchSize=5000/" conf/目标端.properties

# 重启 KFS
bin/flysync restart -dir data/target
```

## 六、手动恢复同步

### 6.1 完全重新同步（最后手段）

```bash
# 1. 停止 KFS
bin/flysync stop -dir data/target

# 2. 清除状态
rm -rf data/target/*

# 3. 重新初始化
bin/flysync init -conf conf/目标端.properties -dir data/target

# 4. 重新启动
bin/flysync start -conf conf/目标端.properties -dir data/target
```

### 6.2 从指定 LSN 恢复

```bash
# 1. 查看可用 LSN
SELECT pg_current_wal_lsn();

# 2. 指定起始 LSN（谨慎使用）
cat >> conf/目标端.properties << 'EOF'
replication.start.lsn=0/12345678
EOF

# 3. 重启 KFS
bin/flysync restart -dir data/target
```

## 七、监控断点续传

### 7.1 监控脚本

```bash
cat > /opt/kingbase-flysync/bin/check_sync.sh << 'EOF'
#!/bin/bash
# 断点续传监控脚本

LOG_FILE="/opt/kingbase-flysync/logs/sync_check.log"

while true; do
    TIMESTAMP=$(date "+%Y-%m-%d %H:%M:%S")
    
    # 检查 KFS 状态
    STATUS=$(bin/flysync status -dir data/target 2>&1)
    
    if echo "$STATUS" | grep -q "Online"; then
        echo "$TIMESTAMP - KFS: Online" >> $LOG_FILE
    elif echo "$STATUS" | grep -q "Recovering"; then
        echo "$TIMESTAMP - KFS: Recovering (断点续传中)" >> $LOG_FILE
    else
        echo "$TIMESTAMP - KFS: Error - $STATUS" >> $LOG_FILE
    fi
    
    sleep 60
done
EOF

chmod +x /opt/kingbase-flysync/bin/check_sync.sh
```

### 7.2 断点续传成功判断标准

1. KFS 状态显示 `Online`
2. `last_wal_receive_lsn` 和 `last_wal_replay_lsn` 在持续更新
3. 源端和目标端数据量一致
4. KFS 日志中无 ERROR 级别错误
