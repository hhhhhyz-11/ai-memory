# 学习总结 - 2026-03-17

## 📚 技术知识

### GitLab 大版本升级路径规划
- GitLab 11.1.4 → 16.1.8 升级需要遵循官方升级路径，不能跨版本升级
- 必须逐级升级：11.1.4 → 11.11.8 → 12.0.12 → ... → 16.1.8（共17个步骤）
- 每个大版本升级都需要完整的中间版本升级链

### Docker 容器化 GitLab 运维
- 使用 Docker 部署 GitLab，通过 volume 映射持久化数据
- 关键数据目录：`/var/opt/gitlab`（数据）、`/var/log/gitlab`（日志）、`/etc/gitlab`（配置）
- 备份命令：`gitlab-rake gitlab:backup:create`
- 恢复命令：`gitlab-rake gitlab:backup:restore BACKUP=<备份名>`

### PostgreSQL 数据库操作
- 登录数据库：`sudo gitlab-rails dbconsole --database main`
- 清理 CI/CD 变量：`DELETE FROM ci_group_variables; DELETE FROM ci_variables;`
- 清除 tokens：更新多个表的 runners_token 和加密字段

### Rails 控制台操作
- 进入 Rails 控制台：`docker exec -it <容器名> gitlab-rails console`
- 删除项目变量：`Project.find_each { |project| project.variables.destroy_all }`

## ⚠️ 踩坑记录

### 1. gitlab-secrets.json 导致 CI/CD 页面 500 错误
- **问题**：旧版本备份中的 `gitlab-secrets.json` 导致所有项目 CI/CD 页面出现 500 错误
- **表现**：页面报错 `aes256_gcm_decrypt`
- **解决**：通过数据库删除所有 CI/CD 变量，或进入 Rails 控制台清除

### 2. GitLab 克隆地址变成容器 ID
- **问题**：升级后 Git 克隆地址变成 `git@7bd63ee2fcc6:inner/data-board.git`（容器ID）而非 IP 地址
- **原因**：旧配置文件中的 external_url 没有正确更新到新容器
- **解决**：修改 `/etc/gitlab/gitlab.rb` 中的 `external_url` 并执行 `gitlab-ctl reconfigure`

### 3. 语言环境不匹配导致 PostgreSQL 启动失败
- **原因**：系统语言环境 `en_US.UTF-8` 与 GitLab 期望的不匹配
- **解决**：确保主机系统安装并配置正确的语言环境

## 💡 经验教训

1. **恢复到相同版本再升级**：不能跨版本恢复备份，必须用相同版本容器恢复后再升级

2. **语言环境必须匹配**：`en_US.UTF-8` 必须匹配，否则 PostgreSQL 启动失败

3. **每版本升级后验证**：建议每个版本升级后验证 2 小时以上再继续下一步

4. **增量备份**：每次升级前都要执行 `gitlab-rake gitlab:backup:create` 增量备份

5. **回滚准备**：删除容器前确保可以用旧镜像重新启动，回滚时用相同版本的镜像

6. **CI/CD 变量问题**：大版本升级时，加密密钥可能不兼容，建议升级前导出变量配置，升级后重新设置
