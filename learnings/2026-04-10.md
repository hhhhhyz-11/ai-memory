# 2026-04-10 学习总结

## 今日重点：KingBase kpatch 原地升级

### 1. kpatch 升级机制

**核心原理**：kpatch 是 KingbaseES 提供的原位补丁升级工具，不需要导出/导入数据，直接替换 Server 程序目录中的二进制文件。

**支持版本范围**：
- PS014 支持从 **PS007** 和 **PS009** 两个基线版本升级
- 不支持跨大版本（如 V9R2C14 不能直接升 PS014）

**升级 vs 回退**：
- `apply single` / `apply cluster_rolling` — 升级
- `rollback single` / `rollback cluster_rolling` — 回退

### 2. 踩坑经验

**坑1：kingbase_path 不能包含 KESRealPro**

```
错误：kingbase_path:/home/kingbase/KESRealPro/V009R002C013/Server contains:KESRealPro
解决：改用软链接路径 /home/kingbase/Server
```

**坑2：数据库必须先启动才能执行 kpatch**

kpatch 的 check 阶段会验证 kingbase 服务是否运行：
```
checking kingbase status 
kingbase not running, please check
→ 必须先 sys_ctl start
```

**坑3：PID 文件残留导致无法启动**

```
错误：lock file "kingbase.pid" already exists
解决：sys_ctl -D .../data stop（清除PID），再 start
```

### 4. KingBase 集群 vs 主备

**主备集群（Data Guard）**：
- 一主一备，流复制同步
- 备机平时只做 HA，不承接读流量
- 故障时 VIP 漂移，备机接管

**读写分离集群（RWC）**：
- 主备基础上，备机主动承接读流量
- 应用层需要配置读写分离路由（KingBase 无透明中间件）
- 多备机可水平扩展读吞吐

**配置文件差异**：两者 install.conf 几乎一样，RWC 多了 `db_mode`、`db_case_sensitive` 等 Oracle 兼容参数。

### 5. tcpdump 抓包分析

**HTTP CONNECT 隧道**：
```
客户端: 192.168.100.60:28003
代理:   192.168.100.125:8080
目标:   rz.weijing.gov.cn:443
```

- `HTTP/1.1 CONNECT rz.weijing.gov.cn:443` = 代理隧道建立请求
- `HTTP/1.1 200 Connection established` = 隧道建立成功，后续全部是加密的端到端流量
- 两次建联 = 第一次 HTTP 代理认证，第二次是实际业务请求

### 6. GitHub 协作经验

- Windows 机器上的 Vagrant 项目，Vagrantfile 在 `F:\virtualbos\`
- WSL 里需要用 `/mnt/f/...` 访问
- GitHub 文件名含中文时，push 后 URL 编码自动转换
