## 一、kpatch 升级核心要点

### 1.1 升级原理

kpatch 是 KingBase 提供的**原位补丁升级工具**，在不重新初始化数据库的情况下直接替换程序文件，完成版本升级。

**关键优势：**
- 无需 sys_dump / sys_restore 全量迁移
- 停机时间短（本次实战 23s）
- 数据目录不变

### 1.2 支持的升级路径

| 基线版本 | 是否可升级到 PS014 |
|---------|-----------------|
| V009R002C013PS007 | ✅ |
| V009R002C013PS009 | ✅ |
| 其他版本 | ❌ 需先升级到 PS007/PS009 |

### 1.3 kpatch 路径参数关键规则

```
❌ 错误：./kpatch -k /home/kingbase/KESRealPro/V009R002C013/Server ...
✅ 正确：./kpatch -k /home/kingbase/Server ...
```

报错信息：`kingbase_path contains:KESRealPro, should not use with:-k *KESRealPro*`

**原因**：安装目录结构中 `KESRealPro` 是版本子目录，kpatch 要求使用软链接路径作为入口。

## 二、升级步骤实战（7 步）

```
1. 确认当前版本        → ./kingbase -V
2. 备份数据目录        → cp -r data/ data_bak/
3. 上传补丁包
4. 解压 kpatch         → tar -zxf ... --strip-components 2 -C .. ./bin/kpatch
5. 启动数据库          → sys_ctl -D data start
6. 执行 kpatch apply   → ./kpatch -k /home/kingbase/Server -D data -t V009R002C013PS014 apply single
7. 验证版本            → ./kingbase -V
```

## 三、常见报错与解决

| 报错 | 原因 | 解决 |
|------|------|------|
| `kingbase not running` | 数据库未启动 | 先 `sys_ctl start` |
| `lock file "kingbase.pid" already exists` | PID 文件残留 | `sys_ctl stop` 后重试 |
| `-k path contains KESRealPro` | 路径含 KESRealPro | 改用软链接路径 |
| `cannot drop operator` (扩展升级) | 有对象依赖操作符 | 先备份/删除依赖对象 |

## 四、版本回退方法

### 4.1 kpatch 回退（保留数据）

```bash
./kpatch -k /home/kingbase/Server -D /home/kingbase/data \
  -t V009R002C013 rollback single
```

### 4.2 数据回退（彻底回退）

```bash
# 停止数据库
./sys_ctl -D /home/kingbase/data stop

# 还原备份的数据目录
rm -fr /home/kingbase/data/
cp -r /home/kingbase/data_bak/data /home/kingbase/

# 启动
./sys_ctl -D /home/kingbase/data start
```

## 五、KingBase 集群 repmgr 常用命令

### 查看集群状态
```bash
./repmgr cluster show          # 整体状态
./repmgr service status        # 含 repmgrd 守护进程
./repmgr cluster show --verbose  # 详细信息
```

### 流复制监控
```sql
SELECT * FROM sys_stat_replication;           -- 复制连接
SELECT * FROM sys_replication_slots;          -- 复制槽
SELECT client_addr, state, sent_lsn, replay_lsn FROM sys_stat_replication; -- 延迟
```

### 启停
```bash
./sys_ctl -D .../data start   # 启动
./sys_ctl -D .../data stop -m fast  # 优雅停止
./sys_ctl -D .../data stop -m immediate  # 强制停止
```

### 故障切换
```bash
./repmgr standby switchover   # 手动切换
./repmgr standby promote       # 提升备库
./repmgr cluster event         # 查看事件日志
```

## 六、HTTP CONNECT 抓包分析

**场景：** 192.168.100.60 通过代理 192.168.100.125:8080 访问外网

关键特征：
- `HTTP: CONNECT rz.weijing.gov.cn:443` → 建立隧道
- `HTTP/1.1 200 Connection established` → 代理允许
- 后续流量变成纯 TCP 隧道（加密的 HTTPS）

**注意：** 普通 tcpdump 抓包只能看到加密流量的存在，无法解密内容。

## 🔑 核心经验

1. **kpatch -k 参数** 永远用软链接路径，不要用 KESRealPro 完整路径
2. **升级前必须启动数据库**，否则 kpatch check 直接失败
3. **PID 文件残留** 是常见启动失败原因，先 stop 再 start
4. **升级前必做数据备份**，即使 kpatch 升级失败也能回退
5. **PS009→PS014 无额外 SQL 操作**，但 PS007→PS009 需要处理 wm_concat 扩展
