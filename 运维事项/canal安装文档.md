## canal安装文档

#### **官方文档要求，binlog格式一定要row模式**

**my.cnf**`# 在my.cnf或my.ini配置如下，开启binlog日志``[mysqld]``log-bin=mysql-bin # 开启 binlog``binlog-format=ROW # 选择 ROW 模式``server_id=1 # 配置 MySQL replaction 需要定义，不要和 canal 的 slaveId 重复``lower_case_table_names=1 # 不区分大小写` `# 说明：广州电动车默认开启的是mixed模式，只能记录原始SQL，这种模式也能监听binlog，但是监听的内容不能解析为ROWDATA，只能业务层面去解析原始SQL再执行业务逻辑`
- #### **为canal创建新的复制账号**

`# 授权 canal 链接 MySQL 账号具有作为 MySQL slave 的权限, 如果已有账户可直接 grant``CREATE` `USER` `canal IDENTIFIED BY` `'canal@yst.com'; ``GRANT` `SELECT, REPLICATION SLAVE, REPLICATION CLIENT ON` `*.* TO` `'canal'@'%';``# 注意，不需要刷新权限的，这里只是写出来而已（很多的教程都是不清楚，所以都加上这句）``FLUSH PRIVILEGES;`

- #### **Canal优化总结**

`** 说明：高屋建瓴的方式来看，我们应该先建立大的局面观，MySQL服务端 -> Canal服务端 -> RabbitMQ -> Canal客户端 -> MySQL目标服务端，所以，优化方向应该也是这五大方面来展开：**``### MySQL服务端优化``1、减少大事务、长事务操作，避免用Navicat等客户端工具去批量复制表，特别的复制大表，这些大表复制出来也会生产binlog日志，导致大量事务日志压垮canal。``2、做数据迁移、数据备份恢复、load命令、mysqldump命令时，应该关闭MySQL的binlog日志``3、优化binlog日志参数，提升刷盘能力和减少CPU占用率。修改事务提交阶段参数`innodb_flush_log_at_trx_commit=2`，避免每个事务执行一次刷盘操作，减少刷盘带来的cpu占用率。` `### Canal服务端优化``1、搭建Canal server HA集群，预防单点故障，支持MHA故障转移，提升并发消费能力``2、调整Canal server对应JVM参数，增大堆内存，减少Full GC``3、调优canal.properties和instance.properties配置，提成并发消费能力``## canal.properties配置优化，官网说了，主要3块优化：1是网络读取优化（sendBufferSize/receiveBufferSize），2是binlog parser优化（parallelBufferSize），3是protobuf的序列化和传输优化（这块待发现）。以下是具体优化参数。``# false代表二进制协议，true代表使用json格式，二进制协议有更好的性能，非flatMessage模式相比于flatMessage模式，大概有30%~60%的性能提升。如果设置为false，MQ收到的消息为protobuf格式，需要通过CanalMessageDeserializer进行解码。``canal.mq.flatMessage=false``# 设置为true表示Canal会以原始数据格式存储数据，即直接从binlog中读取的数据格式。设置为false则表示Canal会转换数据格式，例如转换为JSON格式。针对不开启flatMessage，默认设置canal.instance.memory.rawEntry=true，确保在发送kafka消息阶段不需要做bytes序列化。针对单队列的情况，可以有50%的性能提升``canal.instance.memory.rawEntry=true``# Canal 服务端接收 MySQL 数据包的缓冲区大小，默认16KB。如果 Canal 服务器经常处于等待接收数据的状态，可以适当增加接收缓冲区大小。默认设置的16k的socket buffer，针对大吞吐量传输时有点过小，去掉了默认设置，让socket自我协调，默认在24c96g的物理机上测试，receiveBuffer协调结果为180k.``canal.instance.network.receiveBufferSize=16384``# Canal 服务端发送数据包到 Client 端的缓冲区大小，默认16KB。如果 Canal 需要频繁发送大量数据，可以适当增加发送缓冲区大小。``canal.instance.network.sendBufferSize=16384``# 是否开启binlog并行解析模式(串行解析资源占用少,但性能有瓶颈, 并行解析可以提升近2.5倍+)``canal.instance.parser.parallel=true``# 控制 Canal 在解析 MySQL binlog 时使用的并行线程数``canal.instance.parser.parallelThreadSize=`Runtime.getRuntime().availableProcessors() * 60 / 100; //` `60%的能力跑解析,剩余部分处理网络.```# binlog并行解析的异步ringbuffer环形队列缓冲区的大小(必须为2的指数)，根据 Canal 的官方文档和社区经验，推荐的 canal.instance.parser.parallelBufferSize 值通常在 1024 到 4096 之间。``canal.instance.parser.parallelBufferSize=256``# 是否忽略dcl语句，GRANT、REVOKE语句。``canal.instance.filter.query.dcl=true``# 是否忽略dml语句，增删改查SQL语句。``canal.instance.filter.query.dml=false``# 是否忽略ddl语句，CREATE、ALTER、DROP、TRUNCATE、RENAME语句。``canal.instance.filter.query.ddl=true``# 是否忽略事务头和尾,比如针对写入kakfa的消息时，不需要写入TransactionBegin/Transactionend事件``canal.instance.filter.transaction.entry=true``# 是否过滤掉DML数据变更事件，true：过滤掉DML，只订阅DDL/DCL的操作，false：不过滤DML语句，所有事件都会解析到，源码：LogEventConverter.parseRowsEvent#522行。``canal.instance.filter.rows=false``# MQ消息发送并行度（根据情况调整）``canal.mq.send.thread.size=30``# MQ消息构建并行度（根据情况调整）``canal.mq.build.thread.size=8``# 支持的binlog format格式列表，目前canal支持所有模式的增量订阅，但配合同步时，因为statement只有sql语句，没有数据，所以无法获取原始的变更数据，所以一般建议为ROW模式``canal.instance.binlog.format=ROW,STATEMENT,MIXED``# 是否忽略binlog表结构获取失败的异常(主要解决回溯binlog时,对应表已被删除或者表结构和binlog不一致的情况)``canal.instance.filter.table.error=true` `### RabbitMQ服务端优化``1、开启多实例监听MQ并发消费，开启多个消费实例，处理MQ消息，提升消费速率。``2、搭建RabbitMQ集群模式，预防单点故障``3、配置动态Topic，不同表名发送到不同Topic处理` `### Canal客户端优化``1、开启多线程消费模式，提升消费速率（由于我们消费MQ只做插入逻辑，所以可以不考虑顺序性问题）``2、减少消费时长，将非主业务流程放在异步处理，提升消费速率。``3、开启手动事务管理，减少锁竞争，提升数据库处理能力。修改事务隔离级别，调整为RC模式，减少锁冲突带来的锁等待，消除死锁隐患。``4、增加数据库连接数，spring.datasource.dynamic.druid.max-active = 50，接入Druid连接池复用连接。` `### MySQL目标服务端优化``1、调整innodb buffer pool size。`
