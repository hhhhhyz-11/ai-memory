# KingbaseES V9R2C14 告警规则配置

> 本文档介绍在 KEMCC 控制中心中配置和管理告警规则的完整指南。

---

## 一、告警概述

### 1.1 告警类型

| 告警类别 | 说明 | 示例 |
|---------|------|------|
| 数据库连接 | 数据库可用性告警 | 连接失败、超时 |
| 性能指标 | 性能异常告警 | CPU 高、内存满 |
| 复制状态 | 主备复制告警 | 复制延迟、断开 |
| 存储空间 | 磁盘空间告警 | 表空间满、日志满 |
| 安全告警 | 安全相关告警 | 登录失败、权限异常 |
| 备份状态 | 备份相关告警 | 备份失败、备份延迟 |

### 1.2 告警级别

| 级别 | 标识 | 含义 | 处理方式 |
|------|------|------|---------|
| 紧急 | CRITICAL (P1) | 服务不可用 | 立即处理 |
| 严重 | HIGH (P2) | 功能受损 | 小时内处理 |
| 警告 | WARNING (P3) | 潜在问题 | 计划处理 |
| 信息 | INFO (P4) | 提示信息 | 按需处理 |

## 二、KEMCC 告警规则配置

### 2.1 通过 Web 界面配置

1. 登录 KEMCC 控制台
2. 进入「监控」→「告警管理」
3. 点击「创建告警规则」
4. 配置告警条件、阈值和通知方式

### 2.2 通过配置文件配置

```bash
vim /opt/KEMCC/conf/alert_rules.properties

# ================== 告警规则配置 ==================

# 数据库连接告警
alert.db.connection.enabled=true
alert.db.connection.threshold=3
alert.db.connection.interval=60

# CPU 使用率告警
alert.cpu.enabled=true
alert.cpu.warning=70
alert.cpu.critical=90
alert.cpu.interval=60

# 内存使用率告警
alert.memory.enabled=true
alert.memory.warning=80
alert.memory.critical=95
alert.memory.interval=60

# 磁盘空间告警
alert.disk.enabled=true
alert.disk.warning=80
alert.disk.critical=95
alert.disk.interval=300

# 连接数告警
alert.connections.enabled=true
alert.connections.warning=300
alert.connections.critical=450
alert.connections.interval=60

# 复制延迟告警
alert.replication.enabled=true
alert.replication.warning=100MB
alert.replication.critical=1GB
alert.replication.interval=60

# 慢查询告警
alert.slow_query.enabled=true
alert.slow_query.threshold=5000
alert.slow_query.unit=ms
alert.slow_query.interval=300

# 告警抑制（避免告警风暴）
alert.suppression.enabled=true
alert.suppression.period=300
```

## 三、常用告警规则详解

```properties

SELECT 
    COUNT(*) 
FROM sys_stat_activity 
WHERE state = 'active';

SELECT 1;
```

### 3.5 连接数告警

```properties
# 连接数告警
alert.connections.enabled=true
alert.connections.warning=300            # 当前连接数的百分比阈值
alert.connections.critical=450
alert.connections.interval=60
```

```sql
-- 对应的监控 SQL
SELECT 
    COUNT(*) AS current_connections,
    (SELECT setting::int FROM sys_settings WHERE name = 'max_connections') AS max_connections,
    ROUND(COUNT(*) * 100.0 / (SELECT setting::int FROM sys_settings WHERE name = 'max_connections'), 2) AS usage_percent
FROM sys_stat_activity;
```

### 3.6 复制延迟告警

```properties
# 复制延迟告警
alert.replication.enabled=true
alert.replication.warning=104857600      # 100MB
alert.replication.critical=1073741824    # 1GB
alert.replication.interval=60
```

```sql
-- 对应的监控 SQL（主库执行）
SELECT 
    client_addr,
    state,
    sent_lsn - replay_lsn AS replication_delay_bytes,
    pg_size_pretty(sent_lsn - replay_lsn) AS delay_size
FROM sys_stat_replication;
```

### 3.7 慢查询告警

```properties
# 慢查询告警
alert.slow_query.enabled=true
alert.slow_query.threshold=5000          # 毫秒
alert.slow_query.interval=300
alert.slow_query.severity=INFO
```

```sql
-- 查看当前运行的慢查询
SELECT 
    pid,
    now() - query_start AS duration,
    state,
    query
FROM sys_stat_activity
WHERE state != 'idle'
AND now() - query_start > interval '5 seconds'
ORDER BY query_start;
```

### 3.8 表空间使用率告警

```properties
# 表空间告警
alert.tablespace.enabled=true
alert.tablespace.warning=80
alert.tablespace.critical=95
alert.tablespace.interval=300
```

```sql
-- 查看表空间使用情况
SELECT 
    spcname,
    pg_size_pretty(pg_tablespace_size(spcname)) AS total_size,
    pg_size_pretty(pg_tablespace_size(spcname) - 
        COALESCE(SUM(pg_database_size(d.datname)), 0)) AS used_size,
    ROUND((pg_tablespace_size(spcname) - 
        COALESCE(SUM(pg_database_size(d.datname)), 0)) * 100.0 / 
        pg_tablespace_size(spcname), 2) AS usage_percent
FROM pg_tablespace 
LEFT JOIN pg_database d ON dattablespace = oid
GROUP BY spcname, oid;
```

## 五、告警脚本自定义

### 5.1 自定义告警脚本

```bash
#!/bin/bash
# alert_custom.sh - 自定义告警处理脚本

ALERT_LEVEL=$1
ALERT_MESSAGE=$2
ALERT_HOST=$3
ALERT_TIME=$4

LOG_FILE="/var/log/kemcc_alerts.log"

echo "[$ALERT_TIME] [$ALERT_LEVEL] $ALERT_HOST: $ALERT_MESSAGE" >> $LOG_FILE

case "$ALERT_LEVEL" in
    "CRITICAL")
        # 发送紧急告警（电话、短信）
        curl -X POST "http://ops-system/api/alert" \
            -H "Content-Type: application/json" \
            -d "{\"level\":\"CRITICAL\",\"message\":\"$ALERT_MESSAGE\"}"
        ;;
    "HIGH")
        # 发送高优先级告警（短信、邮件）
        send_sms "$ALERT_MESSAGE"
        send_email "$ALERT_MESSAGE"
        ;;
    "WARNING")
        # 发送普通告警（邮件）
        send_email "$ALERT_MESSAGE"
        ;;
    *)
        # 记录日志
        echo "[$ALERT_TIME] [INFO] $ALERT_MESSAGE" >> $LOG_FILE
        ;;
esac
```

### 5.2 注册自定义告警脚本

```bash
vim /opt/KEMCC/conf/alert_handlers.properties

# 自定义告警处理器
alert.handler.1.type=script
alert.handler.1.path=/opt/scripts/alert_custom.sh
alert.handler.1.severity=CRITICAL,HIGH,WARNING
alert.handler.1.enabled=true
```

## 六、告警管理 SQL

### 6.1 查看告警历史

```sql
-- 查看告警历史
SELECT 
    alert_id,
    alert_name,
    severity,
    instance_name,
    metric_name,
    metric_value,
    threshold,
    message,
    event_time,
    status
FROM kemcc_alert_history
WHERE event_time > NOW() - INTERVAL '24 hours'
ORDER BY event_time DESC;
```

### 6.2 统计告警

```sql
-- 按严重级别统计告警
SELECT 
    severity,
    COUNT(*) AS alert_count
FROM kemcc_alert_history
WHERE event_time > NOW() - INTERVAL '7 days'
GROUP BY severity
ORDER BY 
    CASE severity 
        WHEN 'CRITICAL' THEN 1 
        WHEN 'HIGH' THEN 2 
        WHEN 'WARNING' THEN 3 
        ELSE 4 
    END;

-- 按监控目标统计告警
SELECT 
    instance_name,
    COUNT(*) AS alert_count
FROM kemcc_alert_history
WHERE event_time > NOW() - INTERVAL '7 days'
GROUP BY instance_name
ORDER BY alert_count DESC;
```

### 6.3 确认和处理告警

```sql
-- 确认告警
UPDATE kemcc_alert_history
SET status = 'ACKNOWLEDGED', acknowledged_by = 'admin', acknowledged_at = NOW()
WHERE alert_id = 'alert_001';

-- 关闭告警
UPDATE kemcc_alert_history
SET status = 'RESOLVED', resolved_by = 'admin', resolved_at = NOW(), resolution = '问题已处理'
WHERE alert_id = 'alert_001';
```

## 七、常见问题

### 7.1 告警不触发

**排查**：
1. 检查告警规则是否启用
2. 检查告警阈值是否正确
3. 检查告警通知配置
4. 查看告警日志

```bash
# 查看告警日志
grep -i "alert\|trigger" /opt/KEMCC/logs/kemcc.log | tail -50
```

### 7.2 告警风暴

**问题**：同一问题产生大量重复告警

**解决**：
1. 启用告警抑制
2. 配置告警聚合
3. 设置告警恢复通知

```properties
# 启用告警抑制
alert.suppression.enabled=true
alert.suppression.period=300       # 5分钟内相同告警只发送一次
```

### 7.3 告警通知延迟

**问题**：告警触发后通知延迟

**解决**：
1. 检查网络连通性
2. 调整检测间隔
3. 优化告警处理脚本
