# KingbaseES V9R2C14 主备集群部署

> 本文档介绍 KingbaseES V9R2C14 主备（Primary-Standby）集群的部署与配置，支持自动故障切换。

---

## 一、集群架构概述

### 1.1 集群类型

| 架构类型 | 说明 | 适用场景 |
|---------|------|---------|
| 共享存储主备 | 主备共用一份存储 | 中等可用性需求 |
| 流复制主备 | 通过 WAL 流复制保持数据同步 | 较高可用性需求 |
| RAC Proxy | 代理层实现负载均衡和故障切换 | 高可用需求 |
| KFS 同步 | 文件级同步复制 | 异地容灾 |

### 1.2 环境规划

| 项目 | 主库 (Primary) | 备库 (Standby) |
|------|---------------|----------------|
| 主机名 | kingbase-p | kingbase-s |
| IP 地址 | 192.168.1.101 | 192.168.1.102 |
| 数据目录 | /data/kingbase/data | /data/kingbase/data |
| WAL 目录 | /data/kingbase/wal | /data/kingbase/wal |
| 端口 | 54321 | 54321 |
| 同步方式 | - | stream (流复制) |

## 二、系统环境准备（两台机器）

### 2.1 基础环境检查

```bash
# 两台服务器都需要执行以下操作

# 检查系统版本
cat /etc/os-release

# 检查 hosts 文件
cat /etc/hosts
# 确保包含：
# 192.168.1.101 kingbase-p
# 192.168.1.102 kingbase-s

# 添加 hosts 解析（如没有）
echo "192.168.1.101 kingbase-p" >> /etc/hosts
echo "192.168.1.102 kingbase-s" >> /etc/hosts

# 关闭防火墙
systemctl stop firewalld
systemctl disable firewalld

# 关闭 SELinux
setenforce 0
sed -i 's/SELINUX=enforcing/SELINUX=disabled/g' /etc/selinux/config
```

### 2.2 SSH 互信配置

```bash
# 在主库服务器执行
su - kingbase
ssh-keygen -t rsa -P '' -f ~/.ssh/id_rsa
ssh-copy-id kingbase@kingbase-s

# 测试 SSH 互信
ssh kingbase@kingbase-s "hostname"
```

### 2.3 时间同步

```bash
# 两台服务器都配置 NTP 时间同步
# 安装 chrony（CentOS 7）
yum install -y chrony

# 主库配置为 NTP 服务器
vim /etc/chrony.conf
# 添加：
# allow 192.168.1.0/24
# 本地作为时间源
server 127.127.1.0 iburst

# 启动服务
systemctl start chronyd
systemctl enable chronyd

# 备库配置 chrony 客户端
vim /etc/chrony.conf
# 添加：
# server kingbase-p iburst

# 重启
systemctl restart chronyd

# 验证时间同步
chronyc sources
```

## 三、主库配置

### 3.1 安装 KingbaseES（如未安装）

参考 [单机部署全流程](./01-单机部署全流程.md) 完成主库数据库安装。

### 3.2 修改主库配置文件

```bash
su - kingbase
vim $KINGBASE_DATA/kingbase.conf

# ================== 主库关键配置 ==================

# 监听地址
listen_addresses = '*'
port = 54321

# 最大连接数
max_connections = 500

# 复制相关配置
wal_level = replica          # 必须设置为 replica 或 logical
max_wal_senders = 5          # 最大 WAL 发送进程数（备库数量 + 2）
max_replication_slots = 5    # 最大复制槽数量
wal_keep_size = 128          # 保留的 WAL 文件大小（MB）
hot_standby = on              # 开启热备
hot_standby_feedback = on    # 备库反馈信息给主库

# 流复制连接认证（在 sys_hba.conf 中配置）

# 其他优化参数
shared_buffers = 4GB
work_mem = 64MB
maintenance_work_mem = 1GB
```

### 3.3 配置复制权限（sys_hba.conf）

```bash
vim $KINGBASE_DATA/sys_hba.conf

# 添加以下内容（必须放在 host all all ... 之前）
# TYPE  DATABASE        USER            ADDRESS                 METHOD
# 允许 replication 连接
host    replication     kingbase        192.168.1.102/32         md5
host    replication     system           192.168.1.102/32         md5

# 允许应用连接
host    all             all              192.168.1.0/24           md5
host    all             all              0.0.0.0/0                md5
```

### 3.4 创建复制用户

```bash
# 连接到主库
ksql -U system -d test

# 创建复制专用角色
CREATE ROLE replication_user WITH REPLICATION LOGIN PASSWORD 'ReplPassword123';

# 或者使用 kingbase 用户
CREATE ROLE kingbase WITH REPLICATION LOGIN PASSWORD 'ReplPassword123';
GRANT CONNECT ON DATABASE test TO kingbase;

# 查看用户
SELECT rolname, rolreplication FROM sys_roles WHERE rolreplication = true;

\q
```

### 3.5 创建复制槽（可选）

```bash
# 使用复制槽需要配置 max_replication_slots
ksql -U system -d test -c "SELECT * FROM sys_create_physical_replication_slot('slot_standby1');"

# 查看复制槽
SELECT * FROM sys_replication_slots;
```

### 3.6 重启主库

```bash
# 重启使配置生效
sys_ctl -D $KINGBASE_DATA restart

# 验证 WAL 发送是否开启
ksql -U system -d test -c "SELECT * FROM sys_stat_replication;"
# 应返回空（还没有备库连接）
```

## 四、备库配置

### 4.1 安装 KingbaseES

备库服务器上安装与主库相同版本的 KingbaseES。

```bash
# 创建必要目录
mkdir -p /opt/KingbaseES/V9R2C14
mkdir -p /data/kingbase/data
mkdir -p /data/kingbase/wal

chown -R kingbase:kingbase /opt/KingbaseES
chown -R kingbase:kingbase /data/kingbase
```

### 4.2 使用 pg_basebackup 创建基础备份

```bash
su - kingbase

# 使用 basebackup 从主库拉取数据
pg_basebackup -h kingbase-p -p 54321 -U kingbase -D /data/kingbase/data -R -Xs -P -v

# 参数说明：
# -h: 主库地址
# -p: 主库端口
# -U: 复制用户名
# -D: 备库数据目录
# -R: 自动创建 standby.signal 和 recovery.conf（V12+无需此选项）
# -Xs: 接收 WAL 文件
# -P: 显示进度
# -v: 详细输出

# 如果主库使用 replication 槽：
# pg_basebackup -h kingbase-p -p 54321 -U kingbase -D /data/kingbase/data -R -Xs -P -v -S slot_standby1
```

### 4.3 手动创建 standby.signal

```bash
# 如果 pg_basebackup 没有自动创建，需要手动创建
touch /data/kingbase/data/standby.signal
chown kingbase:kingbase /data/kingbase/data/standby.signal
```

### 4.4 配置备库 recovery.conf

如果是较低版本 KingbaseES（V9），需要配置 `recovery.conf`：

```bash
# 创建或编辑 recovery.conf
vim /data/kingbase/data/recovery.conf

standby_mode = 'on'
primary_conninfo = 'host=kingbase-p port=54321 user=kingbase password=ReplPassword123 application_name=standby1'
restore_command = 'cp /data/kingbase/wal/%f %p'
recovery_target_timeline = 'latest'
# 如果使用复制槽：
# primary_slot_name = 'slot_standby1'

chown kingbase:kingbase /data/kingbase/data/recovery.conf
```

### 4.5 修改备库 kingbase.conf

```bash
vim /data/kingbase/data/kingbase.conf

# 备库监听地址
listen_addresses = '*'
port = 54321

# 备库最大连接数（应大于等于主库）
max_connections = 500

# 热备配置
hot_standby = on

# 备库查询超时
hot_standby_feedback = on
wal_sender_timeout = 60s
wal_receiver_timeout = 60s
```

### 4.6 启动备库

```bash
# 启动备库
sys_ctl -D $KINGBASE_DATA start -l /data/kingbase/data/logfile

# 查看备库日志
tail -f /data/kingbase/data/log/*.log

# 或者
tail -f /data/kingbase/data/logfile
```

## 五、验证集群状态

### 5.1 主库端验证

```bash
# 查看复制连接状态
ksql -U system -d test -c "SELECT * FROM sys_stat_replication;"

# 返回示例：
#  pid  | usesysid | usename | application_name |  client_addr  | client_hostname | client_port |         backend_start         | backend_xmin |   state   |   sent_lsn   |   write_lsn   |  flush_lsn   | replay_lsn
-------+----------+---------+------------------+---------------+-----------------+-------------+-------------------------------+--------------+-----------+--------------+---------------+--------------+-------------
 12345 |    16384 | kingbase | standby1         | 192.168.1.102 |                 |       54321 | 2024-01-15 10:30:00 +08       |              | streaming | 0/30000060  | 0/30000060   | 0/30000060  | 0/30000060

# 查看复制槽
\ksql -U system -d test -c "SELECT * FROM sys_replication_slots;"

# 查看 WAL 发送进程
ps -ef | grep wal sender | grep -v grep
```

### 5.2 备库端验证

```bash
# 查看 WAL 接收状态
ksql -U system -d test -h localhost -p 54321 -c "SELECT * FROM sys_stat_wal_receiver;"

# 返回示例：
#  pid  |  status  |  received_lsn  |  last_msg_send_time  |  last_msg_receipt_time
-------+----------+----------------+----------------------+------------------------
 12346 | streaming | 0/30000060    |                      |

# 检查是否为热备模式
ksql -U system -d test -h localhost -p 54321 -c "SELECT sys_is_in_recovery();"
#  t 表示当前是备库
```

### 5.3 测试数据同步

```bash
# 在主库创建测试数据
ksql -U system -d test -h kingbase-p -c "CREATE TABLE test_repl (id serial, name text);"
ksql -U system -d test -h kingbase-p -c "INSERT INTO test_repl (name) VALUES ('test1'), ('test2');"

# 在备库验证数据同步
ksql -U system -d test -h kingbase-s -c "SELECT * FROM test_repl;"
# 应看到 test1, test2
```

## 六、故障切换

### 6.1 手动切换（主库正常）

```bash
# 在备库执行切换
# 需要有触发器文件或使用 sys_ctl promote

# 方法一：创建 promote 信号文件
touch /data/kingbase/data/failover.trigger

# 方法二：使用 sys_ctl
sys_ctl -D $KINGBASE_DATA promote

# 方法三：备库内执行 SQL
ksql -U system -d test -c "SELECT sys_promote();"
```

### 6.2 主库故障后备库提升

当主库不可用时，将备库提升为新的主库：

```bash
# 在备库执行
sys_ctl -D $KINGBASE_DATA promote

# 确认提升成功
ksql -U system -d test -c "SELECT sys_is_in_recovery();"
# 应返回 f（不再处于恢复/备库模式）
```

### 6.3 重建备库（故障修复后）

原主库修复后，作为新备库重新加入集群：

```bash
# 在新主库（原来的备库）确认 WAL 接收正常

# 在原主库服务器
su - kingbase

# 删除旧数据
rm -rf /data/kingbase/data/*
rm -rf /data/kingbase/wal/*

# 重新拉取数据
pg_basebackup -h <新主库IP> -p 54321 -U kingbase -D /data/kingbase/data -R -Xs -P -v

# 配置并启动
touch /data/kingbase/data/standby.signal
sys_ctl -D $KINGBASE_DATA start
```

## 七、常用运维命令

### 7.1 集群状态查看

```bash
# 主库端
ksql -U system -d test -c "SELECT pid, application_name, client_addr, state, sent_lsn, replay_lsn FROM sys_stat_replication;"

# 备库端
ksql -U system -d test -c "SELECT pid, status, received_lsn, last_msg_receipt_time FROM sys_stat_wal_receiver;"
```

### 7.2 备库延迟监控

```bash
# 主库查看备库延迟
ksql -U system -d test -c "
SELECT 
    client_addr,
    state,
    sent_lsn,
    write_lsn,
    flush_lsn,
    replay_lsn,
    sent_lsn - replay_lsn AS replication_delay
FROM sys_stat_replication;
"

# 查看延迟大小（以字节为单位）
# sent_lsn - replay_lsn 差值越大，延迟越大
```

### 7.3 备库只读验证

```bash
# 备库默认只读，测试写入是否被阻止
ksql -U system -d test -h kingbase-s -c "INSERT INTO test_repl VALUES (99, 'should_fail');"
# 应返回错误：cannot execute INSERT in a read-only transaction
```

## 八、常见问题

### 8.1 备库连接被拒绝

**错误**：`could not connect to primary: FATAL: no pg_hba.conf entry for replication

**解决**：在主库 `sys_hba.conf` 添加：
```
host    replication     kingbase        192.168.1.102/32         md5
```
然后 `sys_ctl reload`

### 8.2 WAL 堆积导致备库延迟

**解决**：
1. 主库：`SELECT sys_checkpoint();`
2. 备库延迟较大时，考虑重新做基础备份
3. 检查网络带宽

### 8.3 备库无法启动

**错误**：`PANIC: could not open required system table

**解决**：检查 `/data/kingbase/data` 目录权限是否为 `kingbase:kingbase`
