# KingbaseES V9R2C14 告警规则配置

> 适用版本：KingbaseES V9R2C14
> 参考：https://docs.kingbase.com.cn/cn/KES-V9R2C14/

---

## 一、告警概述

KingbaseES 提供多种告警方式：
1. **KEMCC 告警** — 通过 KEMCC 平台配置和管理告警
2. **系统视图告警** — 通过查询系统视图获取告警信息
3. **日志告警** — 通过分析日志触发告警
4. **扩展插件告警** — 通过 pg_stat_statements 等扩展监控

---

## 二、常用告警指标

### 2.1 必监控指标

| 指标 | 说明 | 阈值建议 | 严重级别 |
|------|------|---------|---------|
| 连接数使用率 | 当前连接数 / max_connections | > 80% | 警告 |
| 内存使用率 | 数据库内存使用 | > 85% | 警告 |
| 磁盘空间 | 表空间使用率 | > 80% | 警告 |
| 复制延迟 | 主备延迟字节数 | > 1GB | 警告 |
| Long Query | 长时间运行的查询 | > 300s | 警告 |
| Lock Wait | 锁等待事件 | 持续 > 60s | 警告 |
| XLOG 积压 | WAL 日志未归档 | > WAL 保留量 | 严重 |
| 数据库不可用 | 连接失败 | 发生 | 严重 |

## 三、KEMCC 告警配置

### 3.1 登录 KEMCC

```
访问：http://<kemcc服务器>:8080/kemcc
路径：告警 → 告警规则
```

### 3.2 创建告警规则

```sql
-- 在 KEMCC 界面配置以下规则：

-- 1. 连接数告警
规则名称: connection_usage
监控指标: 连接数使用率
条件: > 80%
持续时间: 5 分钟
严重级别: 警告

-- 2. 复制延迟告警
规则名称: replication_lag
监控指标: 复制延迟
条件: > 1073741824 (1GB)
持续时间: 1 分钟
严重级别: 警告

-- 3. 长查询告警
规则名称: long_running_query
监控指标: 查询运行时间
条件: > 300 秒
持续时间: 1 分钟
严重级别: 警告
```

## 四、手动 SQL 监控脚本（无 KEMCC）

### 4.1 连接数告警

```sql
-- 检查连接数使用率
SELECT
    current_setting('max_connections')::int AS max_conn,
    (SELECT count(*) FROM sys_stat_activity) AS current_conn,
    round(
        (SELECT count(*) FROM sys_stat_activity)::numeric /
        current_setting('max_connections')::numeric * 100,
        2
    ) AS usage_percent;

-- 告警触发条件（usage_percent > 80）
```

### 4.2 复制延迟告警

```sql
-- 检查复制延迟
SELECT
    client_addr,
    state,
    sent_lsn,
    replay_lsn,
    pg_wal_lsn_diff(sent_lsn, replay_lsn) AS lag_bytes,
    pg_size_pretty(pg_wal_lsn_diff(sent_lsn, replay_lsn)) AS lag_pretty
FROM sys_stat_replication;
```

### 4.3 长时间运行查询

```sql
-- 检查超过 N 秒的查询
SELECT
    pid,
    usename,
    client_addr,
    state,
    now() - query_start AS duration,
    query
FROM sys_stat_activity
WHERE state != 'idle'
  AND query_start < now() - interval '300 seconds'
ORDER BY duration DESC;
```

### 4.4 锁等待告警

```sql
-- 检查锁等待
SELECT
    l.locktype,
    l.relation::regclass,
    l.mode,
    l.granted,
    a.pid,
    a.usename,
    a.query,
    now() - a.query_start AS wait_duration
FROM pg_locks l
JOIN sys_stat_activity a ON l.pid = a.pid
WHERE NOT l.granted
ORDER BY a.query_start;
```

### 4.5 磁盘空间告警

```sql
-- 检查表空间使用率
SELECT
    spcname,
    pg_size_pretty(pg_tablespace_size(spcname)) AS size,
    pg_size_pretty(
        pg_tablespace_size(spcname) -
        (SELECT sum(pg_relation_size(oid))::bigint
         FROM sys_class WHERE reltablespace = (SELECT oid FROM sys_tablespace WHERE spcname = 'pg_default'))
    ) AS used
FROM sys_tablespace;
```

## 五、告警脚本示例

### 5.1 综合告警检查脚本

```bash
#!/bin/bash
# KingbaseES 综合告警检查脚本
# 配合 cron 实现定时告警

KEMCC_DB="kemcc_db"
KEMCC_HOST="192.168.1.101"
KEMCC_PORT="54321"
KEMCC_USER="system"
ALERT_EMAIL="dba@example.com"

LOG_FILE="/var/log/kingbase_alert.log"

check_connections() {
    # 连接数检查
    RESULT=$(./ksql -h $KEMCC_HOST -p $KEMCC_PORT -U $KEMCC_USER -d test -t -c "
        SELECT round(
            count(*)::numeric / current_setting('max_connections')::numeric * 100, 2
        )
        FROM sys_stat_activity;
    " 2>/dev/null)

    if [ ! -z "$RESULT" ] && [ $(echo "$RESULT > 80" | bc) -eq 1 ]; then
        echo "WARNING: Connection usage ${RESULT}% > 80%" | tee -a $LOG_FILE
    fi
}

check_replication() {
    # 复制延迟检查
    RESULT=$(./ksql -h $KEMCC_HOST -p $KEMCC_PORT -U $KEMCC_USER -d test -t -c "
        SELECT pg_wal_lsn_diff(
            (SELECT max(sent_lsn) FROM sys_stat_replication),
            (SELECT max(replay_lsn) FROM sys_stat_replication)
        );
    " 2>/dev/null)

    if [ ! -z "$RESULT" ] && [ $(echo "$RESULT > 1073741824" | bc) -eq 1 ]; then
        echo "WARNING: Replication lag > 1GB" | tee -a $LOG_FILE
    fi
}

check_long_queries() {
    # 长查询检查
    COUNT=$(./ksql -h $KEMCC_HOST -p $KEMCC_PORT -U $KEMCC_USER -d test -t -c "
        SELECT count(*) FROM sys_stat_activity
        WHERE state != 'idle'
          AND query_start < now() - interval '5 minutes';
    " 2>/dev/null)

    if [ ! -z "$COUNT" ] && [ "$COUNT" -gt 0 ]; then
        echo "WARNING: $COUNT long running queries found" | tee -a $LOG_FILE
    fi
}

# 执行检查
check_connections
check_replication
check_long_queries

echo "--- $(date) ---" >> $LOG_FILE
```

## 六、常见告警及处理

### 6.1 连接数不足

```
告警：Connection usage > 80%
原因：连接数接近 max_connections
处理：
  1. 杀掉空闲连接
  2. 增加 max_connections
  3. 使用连接池（如 PgBouncer）
```

```sql
-- 杀掉空闲连接
SELECT pg_terminate_backend(pid)
FROM sys_stat_activity
WHERE state = 'idle'
AND query_start < current_timestamp - interval '30 minutes';
```

### 6.2 复制延迟

```
告警：Replication lag > 1GB
原因：网络问题 / 备库处理速度慢 / 主库写入太快
处理：
  1. 检查备库负载
  2. 增加备库资源
  3. 优化备库查询
```

### 6.3 磁盘空间不足

```
告警：Disk usage > 80%
原因：数据增长 / 日志积压
处理：
  1. 清理不需要的数据
  2. 扩展磁盘
  3. 添加新表空间
```

## 七、告警阈值建议

| 监控项 | 警告阈值 | 严重阈值 | 说明 |
|-------|---------|---------|------|
| CPU 使用率 | > 70% | > 90% | 系统层面 |
| 内存使用率 | > 80% | > 95% | 系统/数据库 |
| 连接数使用率 | > 70% | > 85% | 预留 15% 给 superuser |
| 复制延迟 | > 100MB | > 1GB | 可能导致数据丢失 |
| 磁盘使用率 | > 75% | > 90% | 包含表空间和 WAL |
| 长查询 | > 60s | > 300s | 可能阻塞其他操作 |
| 锁等待 | > 30s | > 120s | 可能导致死锁 |
| 临时文件 | > 1GB | > 10GB | work_mem 不足 |
| XLOG 积压 | > 3GB | > 10GB | 归档失败 |
