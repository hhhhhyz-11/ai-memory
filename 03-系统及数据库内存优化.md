## 一、内存优化概述

KingbaseES 的性能高度依赖内存配置，合理的内存参数可以显著提升数据库性能。优化原则：
- **操作系统层**：保障数据库能申请到足够内存，防止 OOM
- **数据库层**：合理分配 shared_buffers、work_mem 等核心内存参数

## 二、操作系统层优化

### 2.1 内存参数检查

```bash
# 查看当前内存情况
free -h
cat /proc/meminfo | grep -E 'MemTotal|MemFree|MemAvailable|Buffers|Cached|SwapTotal'

# 查看内存使用详情
cat /proc/meminfo
vmstat 1 5
```

### 2.2 禁用透明大页（Transparent HugePages）

```bash
# 检查当前状态
cat /sys/kernel/mm/transparent_hugepage/enabled
cat /sys/kernel/mm/transparent_hugepage/defrag

# 临时禁用
echo never > /sys/kernel/mm/transparent_hugepage/enabled
echo never > /sys/kernel/mm/transparent_hugepage/defrag

# 永久禁用（CentOS 7）
cat >> /etc/rc.d/rc.local << 'EOF'
if test -f /sys/kernel/mm/transparent_hugepage/enabled; then
    echo never > /sys/kernel/mm/transparent_hugepage/enabled
fi
if test -f /sys/kernel/mm/transparent_hugepage/defrag; then
    echo never > /sys/kernel/mm/transparent_hugepage/defrag
fi
EOF
chmod +x /etc/rc.d/rc.local

# RedHat/CentOS 6 - 修改 grub
# 在 kernel 行末尾添加：transparent_hugepage=never
```

### 2.3 调整 SWAP 策略（swappiness）

```bash
# 查看当前 swappiness
cat /proc/sys/vm/swappiness

# 临时修改（建议设置为 10-60 之间）
sysctl -w vm.swappiness=30

# 永久修改
echo 'vm.swappiness = 30' >> /etc/sysctl.conf
```

### 2.4 调整 OOM Score

```bash
# 为 kingbase 进程设置较低的 OOM优先级，避免被系统杀死
# 获取 kingbase 进程 PID（假设已启动）
pid=$(pgrep -f "kingbase.*data")

# 设置 OOM score 为负数（越低越不会被杀死）
echo -17 > /proc/$pid/oom_score_adj

# 永久设置 - 在 systemd 服务文件中配置
# /etc/systemd/system/kingbase.service
[Service]
OOMScoreAdjust=-500
```

### 2.5 内存分配策略

```bash
# /etc/sysctl.conf 增加
cat >> /etc/sysctl.conf << 'EOF'
# 内存超额分配（KingbaseES 使用 overcommit 机制）
vm.overcommit_memory = 2
vm.overcommit_ratio = 95

# 脏数据回写策略（降低磁盘 IO）
vm.dirty_ratio = 15
vm.dirty_background_ratio = 5
vm.dirty_writeback_centisecs = 500
vm.dirty_expire_centisecs = 3000
EOF

sysctl -p
```

## 三、数据库核心内存参数

### 3.1 内存参数计算公式

```
可用内存 = 服务器总内存 - 操作系统预留 - 其他服务占用

KingbaseES 可用内存分配建议：
├── shared_buffers      = 可用内存的 25%
├── maintenance_work_mem = 可用内存的 10%（最大 2GB）
├── work_mem           = (可用内存 - shared_buffers) / (max_connections * 预估并发比)
├── effective_cache_size = 可用内存的 75%
└── 预留              = 5-10%
```

### 3.2 典型配置示例

#### 场景 A：16GB 内存服务器

```ini
# kingbase.conf
shared_buffers = 4GB                 # 总内存的 25%
effective_cache_size = 12GB           # 总内存的 75%
work_mem = 64MB                      # 根据并发调整
maintenance_work_mem = 1GB
wal_buffers = 64MB
```

#### 场景 B：32GB 内存服务器

```ini
# kingbase.conf
shared_buffers = 8GB
effective_cache_size = 24GB
work_mem = 128MB
maintenance_work_mem = 2GB
wal_buffers = 128MB
```

#### 场景 C：64GB 内存服务器

```ini
# kingbase.conf
shared_buffers = 16GB
effective_cache_size = 48GB
work_mem = 256MB
maintenance_work_mem = 2GB
wal_buffers = 128MB
```

### 3.3 各参数详解

#### shared_buffers

```ini
# 建议值：服务器内存的 20-30%
# 作用：数据库页缓存，用于缓存数据页
# 最小值：128MB
# 注意：KingbaseES 的 MVCC 实现不完全依赖此参数，所以可以相对 PostgreSQL 设置更高比例

shared_buffers = 4GB
```

#### work_mem

```ini
# 建议值：每个排序/哈希操作可用内存
# 作用：ORDER BY、DISTINCT、哈希连接、排序等操作使用
# 注意：这个值是 per-operator，不是 per-query
# 一个复杂查询可能有多个排序，会消耗多倍 work_mem

# 计算公式：
# work_mem = (总内存 * 0.25 - shared_buffers) / (max_connections * 0.3)

work_mem = 64MB
```

#### maintenance_work_mem

```ini
# 建议值：服务器内存的 10%，最大 2GB
# 作用：VACUUM、CREATE INDEX、ALTER TABLE 等维护操作
# 注意：这个值比 work_mem 大是正常的，因为维护操作优先级高

maintenance_work_mem = 1GB
```

#### effective_cache_size

```ini
# 建议值：服务器内存的 75%
# 作用：优化器估算查询成本时使用的缓存大小参考
# 注意：这个不是实际分配的内存，而是优化器的估算值

effective_cache_size = 12GB
```

#### temp_buffers

```ini
# 建议值：默认 8MB 通常足够
# 作用：存储临时表的缓冲区

temp_buffers = 8MB
```

#### wal_buffers

```ini
# 建议值：WAL 文件大小的倍数，通常 64-128MB
# 作用：WAL 预写日志缓冲区

wal_buffers = 64MB
```

## 四、内存优化实战

### 4.1 查看当前内存使用

```sql
-- 查看数据库内存使用情况
SELECT datname,
       pg_database_size(datname) / 1024 / 1024 AS size_mb
FROM sys_database
ORDER BY pg_database_size(datname) DESC;

-- 查看共享内存使用
SELECT name, setting, unit
FROM sys_settings
WHERE name IN ('shared_buffers', 'effective_cache_size');

-- 查看每个连接使用的内存（通过 work_mem 估算）
SELECT count(*) AS total_connections,
       current_setting('work_mem')::bigint AS work_mem_bytes
FROM sys_stat_activity;
```

### 4.2 监控内存使用

```sql
-- 查看当前内存密集型查询
SELECT pid, usename, query,
       (query_exec_count * current_setting('work_mem')::bigint / 1024 / 1024) AS estimated_mem_mb
FROM sys_stat_activity
WHERE state != 'idle'
ORDER BY query_exec_count DESC
LIMIT 10;
```

### 4.3 大内存排序优化

```sql
-- 对于大结果集排序，增加 work_mem
SET work_mem = '512MB';

-- 大表 ANALYZE
ANALYZE VERBOSE my_large_table;

-- 使用 EXPLAIN ANALYZE 查看执行计划
EXPLAIN (ANALYZE, BUFFERS, FORMAT TEXT)
SELECT * FROM my_table ORDER BY some_column;
```

## 七、常见内存问题

### 7.1 OOM (Out of Memory)

```
症状：数据库进程被系统杀死，dmesg 看到 OOM killer 日志
原因：KingbaseES 内存使用超过系统可用内存
解决：
  1. 降低 shared_buffers
  2. 降低 work_mem
  3. 减少 max_connections
  4. 增加 swap
  5. 调低 oom_score_adj
```

### 7.2 内存不足导致排序失败

```
症状：ERROR: could not sort 排序操作失败
原因：work_mem 太小，无法完成排序
解决：
  1. 临时增加 work_mem: SET work_mem = '256MB';
  2. 永久修改 kingbase.conf
  3. 优化查询，减少排序数据量
```

### 7.3 shared_buffers 设置建议

| 内存大小 | shared_buffers | work_mem | maintenance_work_mem | effective_cache_size |
|---------|---------------|----------|---------------------|---------------------|
| 8GB | 2GB | 32MB | 512MB | 6GB |
| 16GB | 4GB | 64MB | 1GB | 12GB |
| 32GB | 8GB | 128MB | 2GB | 24GB |
| 64GB | 16GB | 256MB | 2GB | 48GB |
| 128GB | 32GB | 512MB | 2GB | 96GB |
